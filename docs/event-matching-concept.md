# ご縁会 / イベントマッチング構想

## 目的

このドキュメントは、EnBloom の将来構想である「マッチングアプリ × 街コン」体験を、今後の開発でブレないように保存するためのものです。

現時点では大規模な街コン機能を実装するのではなく、招待コード・紹介者・`introductions` を土台に、将来的に少人数で安心できる出会いの場へ拡張できるようにします。

## 呼び方の方針

EnBloom では、「街コン」という言葉をそのまま前面に出すよりも、プロダクトの世界観に合う表現を検討します。

候補：

- ご縁会
- Bloom Meet
- EnBloom Circle
- 小さなご縁会
- 紹介者イベント

現時点の推奨コンセプト名は **ご縁会** です。

## ご縁会のコンセプト

EnBloom におけるご縁会は、普通の大型街コンではありません。

**紹介者がひらく、少人数の安心できる出会いの場** です。

特徴：

- 招待コードや紹介者に紐づいた小規模イベントとして扱う。
- 4 人、6 人、8 人、12 人程度の少人数を想定する。
- 参加者は事前にプロフィールや出会いの温度感を確認できる。
- 当日いきなり連絡先交換しなくてもよい。
- イベント後にアプリ内で「また話したい」を送れる。
- 相互に「また話したい」になったらマッチする。
- マッチ後に DM が開放される。

## EnBloom らしい価値

ご縁会は、単なるイベント集客機能ではありません。

EnBloom らしさは、以下にあります。

- 紹介者や招待コードの信頼ルートを起点にする。
- 大人数よりも、安心して話せる少人数を優先する。
- プロフィールと出会いの温度感を事前に見られる。
- 連絡先交換を急がず、アプリ内の相互意思確認でマッチする。
- イベント後の「また話したい」から自然に DM へ進む。
- 紹介者単位の小さなコミュニティ運営に発展できる。

## 将来ロードマップ

### Phase A：招待コード本実装

- 紹介者に紐づいた参加ルートを実装する。
- 無制限招待コードを扱えるようにする。
- `profiles.invited_by` / `profiles.invite_code_used` / `introductions` を記録する。

### Phase B：紹介者ごとの参加者一覧

- 自分の招待コードから登録した人を確認できるようにする。
- 紹介者単位の参加者管理を可能にする。
- 信頼ルートを可視化する。

### Phase C：ご縁会イベント機能

- `events` テーブルを追加する。
- `event_participants` テーブルを追加する。
- 紹介者がイベントを作成できるようにする。
- 招待コード経由のユーザーに案内できるようにする。
- 少人数イベントとして運用する。

### Phase D：イベント後マッチング

- `event_after_likes` テーブルを追加する。
- 「また話したい」機能を追加する。
- 相互に「また話したい」になった場合にマッチさせる。
- マッチ後に DM を開放する。

### Phase E：安全・運用強化

- 参加者の年齢確認を整備する。
- 通報・ブロックを強化する。
- イベント参加ルールを整備する。
- キャンセル対応を設計する。
- 主催者権限を設計する。
- 法務確認を行う。

## 今すぐ実装しないこと

以下は、現時点では実装しない方針です。

- イベント作成機能
- イベント決済
- チケット販売
- 大規模街コン運営
- 参加費管理
- 本人確認 / 年齢確認の本実装
- App Store 展開
- ご縁会の本番運用

ただし、今後の設計で拡張しやすいように、招待コード・紹介者・`introductions` の設計を大事にします。

## 招待コードとの設計接続

**招待コード = ただの入場券ではなく、誰のご縁から来たかを記録する参加ルート** です。

ご縁会は、招待コードや紹介者に紐づいた小規模イベントとして扱います。イベント後に「また話したい」を送り、相互ならマッチ / DM につなげる構想ですが、現時点では実装しません。招待コード・紹介者・`introductions` は、この将来構想に接続できる前提で設計を維持します。
