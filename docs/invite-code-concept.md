# 招待コード思想

## 目的

EnBloom における招待コードは、単なる認証・入場制限のための文字列ではありません。

**招待コード = ただの入場券ではなく、誰のご縁から来たかを記録する参加ルート** です。

EnBloom は、知らない人同士を大量に並べるマッチングアプリではなく、信頼できる人の紹介やご縁から始まる出会いを大切にします。招待コードは、その思想をプロダクトとデータ設計の両方で支えるための中核要素です。

## 基本思想

EnBloom の招待コードは、1 人だけが使う使い切りチケットではなく、紹介者に紐づいた参加ルートとして扱います。

例：

- マイペース男が `MYPACE-2026` という招待コードを発行する。
- そのコードを友人、ラジオリスナー、ダンス仲間などに配る。
- 何人でも同じコードで登録できる。
- 登録した全員が「マイペース男からの紹介」として記録される。
- `used_count` により、そのコードから何人登録したか分かる。
- `max_uses = null` の場合は無制限利用として扱う。
- `max_uses` に数値がある場合だけ利用上限を設ける。

この設計により、EnBloom は「どのコードが使われたか」だけでなく、「誰のご縁から参加したか」を継続的に扱えるようにします。

## 招待コード利用時に記録する情報

招待コード利用時には、以下を記録する方針です。

| カラム / 関係 | 役割 |
| --- | --- |
| `invite_codes.created_by` | 招待コードを発行した紹介者を記録する。 |
| `invite_codes.used_count` | そのコード経由で登録した人数を記録する。 |
| `profiles.invited_by` | ユーザーが誰の紹介で参加したかを記録する。 |
| `profiles.invite_code_used` | ユーザーが登録時に利用した招待コードを記録する。 |
| `introductions.introducer_id` | 紹介した人を記録する。 |
| `introductions.introduced_user_id` | 紹介された参加者を記録する。 |

これにより、EnBloom では「誰からの紹介で入ったか」を大切に扱います。

## 差別化としての紹介制

普通のマッチングアプリは、知らない人同士が大量に並ぶ体験になりやすいです。

EnBloom は、信頼できる人の紹介やご縁から始まる体験を重視します。

EnBloom の差別化は以下です。

- 招待制で参加入口を設計する。
- 紹介者に紐づく参加ルートを持つ。
- 誰の紹介で入ったかが残る。
- 大量スワイプではなく、信頼できるご縁を丁寧に扱う。
- 共通の紹介者や参加ルートが安心感になる。
- 将来的に紹介者単位の小さなコミュニティやイベントにつなげられる。

## 実装・設計で大事にすること

- 招待コードを「使い切りチケット」として固定しない。
- `max_uses = null` を無制限利用として扱える設計を維持する。
- `used_count` により、紹介ルートごとの参加者数を把握できるようにする。
- `profiles.invited_by` と `profiles.invite_code_used` を将来の紹介者表示・参加者管理・イベント案内に活用できるようにする。
- `introductions` は、紹介コメントや紹介者単位の信頼ルートを扱う将来拡張の土台として維持する。

## 今すぐ実装しないこと

このドキュメントは思想と設計方針を保存するためのものです。現時点では以下を急いで実装しません。

- 招待者ごとの詳細なコミュニティ管理画面
- 紹介者ランキング
- 招待コード経由の自動課金・報酬設計
- 紹介者コメントの本格運用

ただし、今後の設計で拡張しやすいように、招待コード・紹介者・`introductions` の設計は大事にします。

## ご縁会機能への接続メモ

招待コード・紹介者・`introductions` は、将来の「ご縁会」機能につながる前提で扱います。

- EnBloom は将来的に「マッチングアプリ × 街コン」の思想を取り入れます。
- ただし大型街コンではなく、紹介者がひらく少人数の「ご縁会」として扱います。
- 招待コードや紹介者に紐づいた小規模イベントを想定します。
- イベント後に「また話したい」を送り、相互ならマッチ / DM につなげます。
- 今すぐ実装はしませんが、招待コード・`introductions` 設計は将来のご縁会機能につながる前提にします。
