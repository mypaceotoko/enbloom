# 招待コード思想

## 中核コンセプト

**招待コード = ただの入場券ではなく、誰のご縁から来たかを記録する参加ルート** です。

ConnectBloom は、信頼できる人の紹介や、共通の興味・活動テーマから始まるつながりを大切にします。招待コードは、その思想をプロダクト体験とデータ設計の両方で支える中核要素です。

## 招待コードで記録したいこと

- どのコードで参加したか。
- 誰がそのコードを発行したか。
- どの紹介ルートから参加したか。
- 同じ紹介ルートにどんな活動テーマがあるか。

この設計により、ConnectBloom は「どのコードが使われたか」だけでなく、「誰の活動・紹介のご縁から参加したか」を継続的に扱えるようにします。

## UI での扱い

招待コード入力画面では、コードを単なるチケットではなく、紹介者に紐づいた参加ルートとして説明します。

例:

> 招待コードは1回限りのチケットではなく、紹介者に紐づいた参加ルートです。同じコードで参加した方は、紹介者からのご縁として記録されます。

## 将来の活用

- 紹介ルートごとの小さな交流会。
- 活動ジャンルごとの ConnectBloom Circle。
- 紹介者がひらくコラボ会。
- 募集ボードで、同じ紹介ルート・興味タグの人に届きやすくする。

## 実装方針

現時点では、既存の invite code RPC、管理画面の招待コード発行・削除・無効化、`profiles.invited_by` / `invite_code_used` の保存方針を維持します。DB テーブル名や RPC 名の大規模リネームはまだ行いません。

## 法務メモ

招待コードは活動・興味・共創・紹介の参加ルートとして扱います。ただし、法的判断は別途専門家確認が必要です。
